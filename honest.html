PROBLEM SOLVING

Netty System Bottleneck — Diagnosis & Resolution

Netty 기반 금융 연동 서버가 높은 동시성 환경에서 간헐적인 레이턴시 스파이크를 경험하고 있었습니다.

가설: 이벤트 루프 스레드가 동기 작업에 의해 블로킹되어, 같은 이벤트 루프를 공유하는 모든 채널에서 Head-of-Line 블로킹이 발생하고 있었습니다.

조사: 스레드 덤프와 JFR 레코딩을 프로파일링하여 이벤트 루프 스레드가 시간을 소비하는 지점을 추적했습니다. 특정 코덱 핸들러가 이벤트 루프 스레드에서 직접 블로킹 I/O(데이터베이스 조회)를 수행하고 있는 것을 발견했습니다.

해결: Netty의 EventExecutorGroup을 사용하여 블로킹 작업을 전용 스레드 풀로 오프로드하고, 이벤트 루프는 순수하게 논블로킹 상태를 유지하도록 했습니다. 부하 테스트로 검증한 결과 — P99 레이턴시가 크게 감소했고, 지속적인 부하 하에서 처리량이 증가했습니다.

TCP Performance Tuning — Financial Protocol Optimization

금융 연동 프로토콜은 고정 길이 메시지 포맷(10바이트 길이 헤더, EUC-KR 인코딩)을 사용하여 Raw TCP로 통신했습니다.

문제: 서버 처리는 빠른데도 불구하고 응답 시간이 일관되지 않았으며, 일부 거래가 예상보다 2-3배 오래 걸렸습니다.

근본 원인: TCP Nagle 알고리즘이 작은 패킷을 배치하고 있었고, 수신 측의 Delayed ACK이 문제를 가중시켰습니다. 또한 소켓 버퍼 사이즈가 금융 프로토콜에서 일반적인 메시지 크기(10-14KB)에 최적화되지 않은 기본값이었습니다.

해결: Nagle 알고리즘 비활성화(TCP_NODELAY), SO_RCVBUF/SO_SNDBUF를 일반적인 메시지 크기에 맞게 튜닝, 장기 유지 금융 연결을 위한 connection keep-alive 파라미터를 설정했습니다. 현실적인 조건에서 개선 사항을 검증하기 위해 TCP 모킹 서버(tcp-css-loadtest)를 직접 구축했습니다.

Load Testing Strategy — Realistic Traffic Simulation

BaaS 모델의 프로덕션 배포 전에 시스템 용량을 검증하고 한계점을 식별해야 했습니다.

접근법: Locust(Python 기반, 복잡한 사용자 시나리오 시뮬레이션)와 k6(Go 기반, Raw 프로토콜 레벨 부하 생성) 두 가지를 활용했습니다. 실제 트래픽 패턴을 반영한 테스트 시나리오를 설계했습니다 — 피크 시간대, 점진적 증가, 급격한 스파이크 버스트.

주요 발견: 지속적인 부하 하에서의 커넥션 풀 고갈, GC pause가 P99 레이턴시에 미치는 영향, 스레드 풀 포화 임계값을 식별했습니다. 각 발견은 타겟팅된 최적화로 이어졌고, 반복적인 벤치마크 사이클을 통해 측정 가능한 개선을 검증했습니다.

JVM / K8s Observability — Visibility into Production

Kubernetes에서 JVM 애플리케이션을 깊은 관측성 없이 운영하는 것은 눈을 감고 비행하는 것과 같았습니다 — 애플리케이션 레벨 메트릭(GC 동작, 스레드 상태, 힙 사용량)과 인프라 레벨 시그널(Pod 재시작, 리소스 제한, 네트워크 레이턴시)을 상관시킬 수 없었습니다.

JVM 내부(JFR, MBeans, GC 로그)와 Kubernetes 메트릭(리소스 사용률, Pod 라이프사이클 이벤트)을 연결하는 관측성 스택을 구축했습니다. 사용자에게 영향을 미치기 전에 이상 징후를 감지할 수 있는 대시보드와 알림 규칙을 만들었습니다 — OOM Kill을 예측하는 GC 압력 트렌드나, 타임아웃 캐스케이드에 선행하는 스레드 풀 고갈 패턴 등.
